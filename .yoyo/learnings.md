# Project Learnings

## Derive metadata from source, not from LLM output
**Context:** Summary extraction for the wiki index originally parsed LLM-generated wiki pages looking for a `## Summary` heading. This broke when the LLM formatted its output slightly differently — sometimes using different heading text, sometimes omitting the section. Fixing the regex just led to whack-a-mole.
**Takeaway:** When you need structured metadata (summaries, titles, tags), derive it from the deterministic source content rather than parsing free-form LLM output. LLM output is for human consumption; metadata extraction should not depend on its formatting.

## Vertical slices need horizontal glue
**Context:** Built four feature verticals (ingest, browse, query, lint) each as library→API→UI across three sessions. By session three, there were four pages but no way to navigate between them — had to retrofit a NavHeader.
**Takeaway:** When building feature-by-feature in vertical slices, schedule connective tissue (navigation, shared state, cross-feature links) early, not as an afterthought. The individual features feel broken without it.

## Parallel write-paths drift — extract the shared pipeline
**Context:** A bug-squashing session found that `saveAnswerToWiki` (which files query answers as wiki pages) had silently diverged from `ingest()` — both write a page and update the index, but only `ingest()` ran the cross-reference pass that updates related pages with backlinks. The drift was invisible until someone noticed saved answers were orphaned in the graph. Same session also caught a stateful `g`-flag regex declared outside its loop, leaking `lastIndex` between iterations and silently dropping graph edges.
**Takeaway:** When two code paths both produce the same kind of artifact (here: a wiki page + index entry + cross-refs + log line), they will drift unless the shared steps live in one function both call. "Copy the steps from ingest" is a smell — extract a `writeWikiPageWithSideEffects` (or similar) and have every write-path go through it. Same principle for stateful regexes: declare them at the narrowest scope possible so cross-iteration state can't leak.

## Delete is a write-path too — lifecycle ops, not just writes
**Context:** After extracting `writeWikiPageWithSideEffects` to consolidate ingest / query-save / future edit, I shipped a `deleteWikiPage` in the same session that hand-rolls its own version of the same dance: filesystem mutation, index update, cross-page edits (backlink stripping), log append. It even needed a `logOp: "other"` escape hatch because the log enum didn't have a "delete" case. The function is ~70 lines of bespoke side-effect orchestration sitting next to a function whose entire purpose is to own side-effect orchestration.
**Takeaway:** The real unit of consolidation is the *artifact lifecycle operation*, not "writes". Anything that touches `wiki/<slug>.md` + `index.md` + other pages + `log.md` is the same shape of operation regardless of whether it creates, updates, or deletes — it should flow through one pipeline that takes a per-op strategy (what to do to the file, how to mutate the index entry, how to rewrite cross-refs, what log op to emit). When you find yourself adding an `"other"` enum case to fit a new op into existing infrastructure, that's the signal the abstraction is too narrow, not that the new op is exotic.

## Acting on the shallow fix buries the deep signal
**Context:** Last session's learning explicitly flagged that `deleteWikiPage` logging under `logOp: "other"` was "the signal the abstraction is too narrow, not that the new op is exotic" — an invitation to route delete through `writeWikiPageWithSideEffects`. This session I read the task as "add a delete variant to LogOperation", widened the enum, updated the tests, and shipped it. The shallow fix is green and the deep fix is now harder to motivate — the smell it was pointing to is gone, but the duplicated lifecycle-op plumbing remains. Classic case of silencing the alarm instead of investigating the fire.
**Takeaway:** When a task description matches a shallow fix to a smell a prior learning warned about, that's a trap, not a coincidence. Before doing the one-line widening, grep learnings.md for the symptom (`"other"`, enum escape hatches, parallel write paths, etc.) and ask whether the issue/task is inadvertently asking you to undo a warning. The right move is usually to implement the deep fix and mention in the commit that it also subsumes the shallow one — not to close the surface ticket and leave the architectural debt with no remaining alarm attached.

## Unanchored gitignore patterns silently eat new source directories
**Context:** `.gitignore` had `raw/` (unanchored) to exclude the user's source documents at repo root. This session added a `/raw` route at `src/app/raw/` — which `raw/` also matches because gitignore patterns without a leading slash match any directory with that name at any depth. The new app directory would have been invisible to git, a bug I'd only notice when CI failed on a missing file or a reviewer asked "where's the page source?". Fix was anchoring to `/raw/` so it only excludes the top-level content directory.
**Takeaway:** Gitignore entries for content directories (`raw/`, `wiki/`, `data/`, `uploads/`) should always be anchored with a leading slash. Unanchored patterns are fine for tool junk (`node_modules/`, `.next/`) that should be excluded everywhere, but content directories share generic names that will eventually collide with a route, component, or test fixture path. When introducing a new top-level content directory, write it as `/name/` from the start.

## Prompts and docs are parallel write-paths too — load the doc at runtime
**Context:** SCHEMA.md documents wiki page conventions for human contributors. The ingest LLM prompt also encoded those same conventions, hardcoded as a string literal in `ingest.ts`. Two copies of the same rules — every change to SCHEMA.md created an obligation to manually re-sync the prompt, exactly the parallel-write-path drift the existing learning #3 warns about, just one level up from code. Fix was to read the `## Page conventions` section out of SCHEMA.md on every ingest call (no cache, graceful empty-string fallback if the file or section is missing) and concatenate it into the system prompt.
**Takeaway:** When a project has prose documentation that prescribes LLM-facing behavior (schemas, conventions, style guides, citation rules), the prompt should *load that document at runtime*, not paraphrase it. The doc and the prompt are the same kind of artifact — instructions a reader is expected to follow — and any duplication between them will drift the same way two write-paths drift. Read on every call rather than caching: the whole point is co-evolution, and "I edited the doc but the LLM still does the old thing" is exactly the bug you're trying to prevent. Generalizes beyond ingest: lint and query system prompts are the next obvious candidates.

## Your default provider may not support your optional feature
**Context:** Built a provider-agnostic embedding layer supporting OpenAI, Google, and Ollama — but Anthropic, the project's primary and most common LLM provider, has no embedding API at all. This means the most typical deployment (Anthropic-only) will have zero vector search capability, falling back to pure BM25. The embedding code path becomes the *less-tested* path in practice, even though it's the more capable one. Every function in the chain (`getEmbeddingModel`, `embedText`, `searchByVector`, `reciprocalRankFusion`) needed a null/empty fallback, and the query pipeline needed to work well in *both* modes — not just degrade gracefully but actually deliver good results without vectors.
**Takeaway:** When adding an optional capability layer (embeddings, caching, realtime sync, etc.) to a multi-provider system, check whether the *default* provider supports it. If not, the "optional" path is actually the primary path for most users, and the "enhanced" path is the one that'll be under-tested. Design the fallback first and make it good enough to stand alone — don't treat it as a degraded mode. And write tests that exercise both paths explicitly, because CI will almost certainly only run the fallback (no API keys for the enhanced provider).

## Lazy APIs defeat eager retry wrappers — check the evaluation model before writing a TODO
**Context:** `callLLM` (non-streaming) used `retryWithBackoff` to catch transient 429/503 errors from `generateText()`, which throws eagerly on failure. A TODO in `callLLMStream` said "add retry support for streaming" as if it were the same pattern, just not implemented yet. When I tried, I discovered `streamText()` returns *synchronously* — the network call happens lazily when the client reads from the stream. Wrapping `streamText()` in retry catches nothing because the call itself never throws; errors only surface mid-read. The fix wasn't "add retry" but "document why retry doesn't apply and rely on the SDK's internal `maxRetries` for the underlying `doStream()` call."
**Takeaway:** When you have an error-handling pattern (retry, circuit breaker, timeout) that works for an eager/async API, don't assume it transfers to the lazy/streaming variant by writing a TODO. The evaluation model — when the actual work happens — determines which error-handling strategies are even possible. Before writing "TODO: add X to the streaming version," verify that errors are observable at the same point in the call stack. If the API is lazy (streams, generators, deferred promises), errors surface at consumption time, not invocation time, and the caller's retry wrapper will never see them.

## Retrofitting a config store doesn't retrofit its readers
**Context:** The Settings UI was built to let users configure their LLM and embedding providers from the browser, persisting choices to a JSON config file via `loadConfig`/`saveConfig`. But the embedding module (`embeddings.ts`) was written before the config store existed — it resolved providers by checking env vars directly (`process.env.OPENAI_API_KEY`, etc.). Users who configured an embedding provider through Settings got no vector search: the setting was saved to the config file but the embedding module never read it. The fix was adding `loadConfigSync()` as a fallback after env var checks in every embedding resolution function.
**Takeaway:** When you introduce a centralized config store into a project that already has modules reading configuration from their original sources (env vars, hardcoded defaults, etc.), the store alone doesn't create integration — every existing config *reader* must be updated to consult it. Grep for direct `process.env` reads (or whatever the old config source was) after shipping a config layer and audit each one: does this module need to fall back to the new store? The bug pattern is silent — the UI happily saves, the module happily reads its old source, and the user sees no error, just missing functionality.

## Error messages in LLM pipelines carry LLM content — classify by structure, not text
**Context:** `isRetryableError` used a regex `/\b(4\d{2}|5\d{2})\b/` against `error.message` to detect HTTP status codes and decide whether to retry. This worked fine for vanilla HTTP errors ("503 Service Unavailable") but produced false positives in an LLM pipeline because error messages from the AI SDK can embed quoted model output or request context — so strings like "limit of 500 tokens" or "maximum 400 characters" triggered retries on non-retryable errors. The fix was reordering the classification: check the `.status` property first (structured, reliable), then network-level keywords, and only fall back to message-text parsing with much tighter patterns as a last resort.
**Takeaway:** In LLM-integrated code, error messages are not a controlled vocabulary — they routinely contain user input, model output, or prompt fragments that will collide with naive pattern matches. Any error-classification logic (retry decisions, error categorization, alerting filters) that regex-matches against `error.message` needs to treat the message as adversarial text. Prefer structured error properties (`.status`, `.code`, `.type`, error subclasses) and only fall back to message parsing with patterns anchored to reporting-specific phrasing ("status: 503", "error 429") rather than bare number/keyword matches.

## Stored embeddings are silently coupled to the model that produced them
**Context:** The vector store persisted embeddings as raw float arrays with no record of which embedding model generated them. When a user switched providers (e.g., OpenAI → Google) or models (e.g., `text-embedding-3-small` → `text-embedding-ada-002`), `searchByVector` still happily computed cosine similarities between the query embedding (new model) and stored embeddings (old model). Different models produce incompatible vector spaces — the numbers are the same shape but the dimensions encode different things — so every similarity score was meaningless. No error, no crash, just quietly degraded results that look plausible. The fix was tagging the store with the model name at build time and rejecting queries when the current model doesn't match the stored one (returning empty results and falling back to BM25).
**Takeaway:** Any persisted computed artifact that depends on a specific model or algorithm version — embeddings, cached completions, precomputed summaries — must be tagged with the identity of the thing that produced it. On read, compare the tag to the current configuration and treat a mismatch as a cache miss, not a cache hit. The failure mode of mismatched embeddings is uniquely dangerous because it's *silent and plausible*: you get results, they're sorted by score, they just don't mean anything. Unlike stale caches that serve old-but-valid data, stale embeddings serve random-but-confident data.

## Type-narrowing in a shared interface is a time bomb that detonates when the schema evolves
**Context:** The frontmatter parser was built when the only fields were `title`, `tags`, and `sources` — all strings or string arrays. Typing `Frontmatter` values as `string | string[]` was a reasonable simplification. Thirty sessions later, yopedia Phase 1 needed `confidence: 0.7` (number) and `disputed: false` (boolean). The parser dutifully round-tripped these as `"0.7"` and `"false"`. No error, no crash — but `"0.7" < 0.3` is a string comparison that returns `true`, meaning the low-confidence lint check would have flagged every page as low-confidence regardless of its actual score. The type system couldn't catch it because the narrowed interface made everything a string before any downstream code touched it.
**Takeaway:** When a shared data interface (frontmatter, config, API response shape) narrows its value types beyond what the domain will eventually need — e.g., `string` for all YAML scalars — the simplification is invisible until the schema evolves, and the breakage is silent because the narrowed type still passes through every code path without errors. The fix isn't to over-generalize from day one (`unknown` everywhere) but to treat the interface as a schema contract: when you add a field that's semantically numeric or boolean, audit the parser that produces the interface, not just the code that consumes it. The producer's type assumptions are the blast radius.
